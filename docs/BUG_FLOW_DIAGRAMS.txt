DIAGRAM 1: ZOMBIE STREAM BUG (Frontend Abort Controller Issue)
===============================================================================

Normal Flow (Correct):
  sendChat()
    │
    ├─→ _chatAbortController = new AC()
    ├─→ fetch('/api/chat', {signal: AC.signal})
    │    │
    │    └─→ Stream starts
    │
    └─→ [User presses Stop]
         │
         └─→ stopChat()
              └─→ _chatAbortController.abort()
                  └─→ Stream canceled immediately


BUGGY Flow (Missing Abort in Switch):
  sendChat()
    │
    ├─→ _chatAbortController = new AC#1
    ├─→ fetch('/api/chat', {signal: AC#1.signal})
    │    │
    │    └─→ Stream starts (5-10 second response expected)
    │
    └─→ [User clicks another session]
         │
         └─→ switchChatSession("Session-B")
              │
              ├─→ fetch('/api/chat/sessions/B/switch')  ← NO stopChat()!
              │
              └─→ [Render Session B]
                   │
                   └─→ AC#1 is STILL ACTIVE
                        │
                        └─→ [5 seconds later...]
                             Stream completes
                             _chatStreaming = false (too late!)
                             AC#1 = null (too late!)
                             
                        ← Memory leak: AC#1 was kept alive


Impact Timeline:
  T=0s    │ Start stream (AC#1)
  T=0.5s  │ Switch session
  T=1s    │ DOM updated (Session B visible)
  T=5s    │ Stream completes (AC#1 finally cleaned)
          │
          ├─ Browser memory: AC#1 object still in memory 0-5s
          ├─ User experience: Session B renders, stream continues invisibly
          └─ Repeated switches: AC pool accumulates unbounded


DIAGRAM 2: RACE CONDITION BUG (Backend Session Swap)
===============================================================================

THREAD TIMELINE:

Thread A: switch_chat_session("/api/chat/sessions/B/switch")
│
├─ T=0ms: _chat_history.clear()
│         _chat_history = []
│
├─ T=1ms: _chat_session_id = "Session-B"
│
└─ T=2ms: messages = await _load_chat_history()
│         (loads Session B: 50 messages)
│

Thread B: POST /api/chat (concurrent)
│                                    ← Starts at T=0.5ms
├─ T=0.5ms: Read _chat_history
│           Result: [] (empty! just cleared by Thread A)
│
├─ T=0.6ms: Append user message
│           _chat_history = [{"role": "user", "content": "Status?"}]
│
├─ T=0.7ms: start_messages = list(_chat_history[-10:])
│           = [{"role": "user", "content": "Status?"}]
│           ← NO PRIOR CONTEXT
│
├─ T=0.8ms: stream_mas_chat(start_messages, ...)
│           MAS starts with: [user_msg_only]
│           (ignores Session B's 50 prior messages)
│
└─ T=5ms:   Complete stream with wrong context

Thread A (resumed):
│
└─ T=2ms:   _chat_history.extend([Session B's 50 messages])
            _chat_history now populated
            BUT: Thread B stream already started with wrong context


Result of Race:
  ✗ Chat lacks prior context
  ✗ MAS thinks session just started
  ✗ Agent response is generic (not domain-aware)
  ✗ User experience: AI seems to have forgotten conversation history


Solution: Add Lock
  @app.post("/api/chat/sessions/{session_id}/switch")
  async def switch_chat_session(session_id: str):
      global _chat_session_id
      with _chat_lock:  ← Thread B blocks here if Thread A running
          _chat_history.clear()
          _chat_session_id = session_id
          messages = await _load_chat_history()
          _chat_history.extend(...)
      return {...}


DIAGRAM 3: ORPHANED ABORT CONTROLLER ACCUMULATION
===============================================================================

Session Switch Sequence (Multiple Times):

Iteration 1:
  ├─ T=0s:   sendChat() → AC#1 = new AbortController()
  ├─ T=2s:   Stream running
  ├─ T=2.5s: switchChatSession("B")  ← AC#1 NOT aborted
  ├─ T=5s:   AC#1 completes, cleaned by sendChat cleanup
  └─ AC#1 lifespan: 5 seconds (correct)

Iteration 2:
  ├─ T=5.5s: sendChat() → AC#2 = new AbortController()
  ├─ T=6.2s: switchChatSession("C")  ← AC#2 NOT aborted
  ├─ T=10s:  AC#2 completes
  └─ AC#2 lifespan: 4.5 seconds (correct)

Iteration 3:
  ├─ T=10.5s: sendChat() → AC#3 = new AbortController()
  ├─ T=10.7s: switchChatSession("D")  ← AC#3 NOT aborted!
  │           BUT stream has only 0.2s elapsed
  │           Expected response: 5 seconds
  ├─ T=11.5s: AC#3 still running (stream in background)
  │           User sends another message → AC#4 created
  │           Now TWO AbortControllers active!
  ├─ T=15.7s: AC#3 finally completes
  └─ AC#3 lifespan: 5.2 seconds (but in background)

Iteration 4+:
  └─ AbortController count: 1, 1, 2, 3, 4, ...
     Memory impact: Grows linearly with rapid switching


Browser Memory Over Time:
  
  Switch Frequency: Every 0.5s
  
  Time      Active ACs    Memory
  ----      -----------   ------
  T=0s      1             ~16KB
  T=5s      1             ~16KB
  T=5.5s    1 (AC#1 released)
  T=10s     1             ~16KB
  T=10.5s   2             ~32KB
  T=11s     2             ~32KB
  T=15.7s   1 (AC#3 released)
  T=16s     1             ~16KB
  
  ← OSCILLATES unless switches slow down
  ← Each miss to abort = 5 extra seconds of memory hold


DIAGRAM 4: COMBINED IMPACT (Both Bugs Active)
===============================================================================

Real User Scenario:

1. Session A: "Tell me about all vehicles"
   └─ Large response → 10 seconds streaming

2. T=2s: User switches to Session B
   ├─ Bug #1: AC#1 NOT aborted → still streaming
   └─ Bug #2: _chat_history cleared (empty)

3. T=2.5s: User sends message to Session B
   ├─ Bug #2: Race condition
   │  _chat_history is empty or partially loaded
   │  start_messages = [user_msg_only]
   ├─ MAS receives: [{"role": "user", "content": "What else?"}]
   │  No prior context from Session B
   └─ MAS response: "I don't have context about your previous questions"

4. T=12s: AC#1 finally completes
   ├─ Large response (~1000 tokens) is discarded
   ├─ _chatStreaming still false (already was set)
   └─ Browser memory: AC#1 held for 10 seconds unnecessarily

User Experience:
  ├─ Session B feels disconnected (lost context)
  ├─ App feels slow (background stream running)
  ├─ After many switches: browser stutters (memory pressure)
  └─ Chat quality: Poor (MAS lacks context)


DIAGRAM 5: ABORT CONTROLLER LIFECYCLE
===============================================================================

Current (Buggy):
  
  sendChat()
    │
    ├─ T=0: _chatAbortController = new AC()
    │
    ├─ T=0: fetch('/api/chat', {signal: AC.signal})
    │
    ├─ T=5: [Stream completes]
    │       _chatAbortController = null
    │       _chatStreaming = false
    │
    └─ AC cleaned up at stream END (correct)
  
  switchChatSession()  ← NO INTERACTION
    │
    ├─ T=2: fetch('/api/chat/sessions/B/switch')
    │       (AC is STILL active if sendChat() running)
    │
    └─ T=2: Render Session B (AC unchanged)
  
  PROBLEM: If sendChat() running at T=2, AC not aborted


Fixed (Correct):

  sendChatFunction()
    │
    ├─ T=0: _chatAbortController = new AC()
    │
    ├─ T=0: fetch('/api/chat', {signal: AC.signal})
    │
    ├─ T=5: [Stream completes]
    │       _chatAbortController = null
    │       _chatStreaming = false
    │
    └─ AC cleaned up at stream END
  
  switchChatSession()  ← CANCELS STREAM
    │
    ├─ T=2: if (_chatAbortController) _chatAbortController.abort()
    │       AC.abort() is called
    │
    ├─ T=2: fetch('/api/chat/sessions/B/switch')
    │       Stream is now aborted (signal triggered)
    │
    ├─ T=2: Render Session B
    │       AC = null
    │       _chatStreaming = false
    │
    └─ AC cleaned up immediately (no dangling refs)
